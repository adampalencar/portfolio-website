## Atmosphere and Clarity Want Opposite Things

How do you make a UI that looks like it belongs in an old D&D manual without burying the numbers that keep players alive?

Isles of Krom is a dark fantasy extraction roguelike. Players guide disposable characters called Wards through hostile, shifting islands, looting gear to strengthen future runs. The loop is fast, and **every screen competes for the player's limited attention.**

If the UI buries stats in texture, **a player sends an underequipped Ward into a run and loses 20 minutes of progress.**

The design system was tight enough that **developers could implement screens after handoff without guessing.** Playtesters in the studio's Discord started calling out the UI unprompted, commenting on the visual identity and how readable the screens felt during play.

## Atmosphere Lives in Structure, Not Texture

Early on I established a framing system built around rounded, two-tone panels. They group related information the same way a well-designed book uses page layout and margins to guide your eye.

The panels create consistent spatial rhythm across every screen. **Players learn the visual grammar once, and it transfers everywhere:** inventory, crafting, merchant, character prep. The "book" feeling comes from this structural choice. Parchment tones and weathered edges help, but the panels carry it.

The cost was rigidity. A strict panel system means less freedom for one-off layouts. Some screens would have been more visually striking with a bespoke composition. But in a roguelike where players cycle through screens quickly between runs, **predictability is a feature.** A player who just lost a Ward and is prepping the next one should not have to relearn the layout.

<div class="cs-callout">

**Design decision:** I chose rigidity over flexibility. Every screen uses the same framing system. The atmosphere comes from the structure repeating, not from any individual screen being visually unique.

</div>

## The Ward Screen Answers One Question: Is This Character Ready?

The Ward preparation screen is the commitment moment. The player is about to send a character into danger, and this is the last chance to evaluate whether they will survive. The layout needed to answer one question instantly: *Is this Ward ready?*

I structured the hierarchy so survivability stats and risk modifiers sit at the top of the visual stack. Equipment and secondary details live below, accessible but never competing for first attention. **The go/no-go read happens in under two seconds.**

This meant deprioritizing information that felt important in isolation: loadout details and lore flavor. **If it does not affect the decision to launch, it waits.** Players who want depth can drill into secondary states, but the survival read comes first.

<Image src={prepareImg} alt="Annotated Ward preparation screen showing survivability stats at the top of the visual stack with equipment and secondary details below" />

The 10 tooltip variants exist because of this screen. Different stats need different levels of explanation: a quick hover for a familiar modifier, a richer tooltip for an unfamiliar debuff. The tooltip system is how the hierarchy works without hiding information.

## Safe Screens Earn Complexity. Run Screens Do Not.

During a run, the player is exploring and fighting. Any UI that appears mid-run is an interruption. The chest interaction screen shows inventory capacity and immediate actions.

This was a bet against information density. Other extraction games surface more data during loot moments: item comparison overlays, detailed stat breakdowns, loot history. I went minimal because **the game's tempo rewards quick decisions, and every extra element on screen is a reason to hesitate.**

<Image src={chestTooltipsImg} alt="In-run chest UI showing the minimal modal, HUD icons, and equipment slots overlaid on the game world" />

The merchant and crafting screens can afford complexity because the player is in a safe state. Browse without losing context, compare items side by side. The run UI cannot afford any of that.

<div class="cs-callout">

**Design decision:** No item comparison overlays, no detailed stat breakdowns, no loot history in the run UI. Those belong in the safe state. The run is for deciding, not studying.

</div>

## The Book Metaphor Holds. The System Is Still Evolving.

The panel system outlasted everything else. Every time the art team added a new environment or the design team changed a mechanic, the framing held. Screens got redesigned multiple times as systems evolved during development. **A solid kit of reusable pieces meant those redesigns took hours instead of days.**

If I could go back, I would have pushed for interactive prototyping earlier. Static mockups communicated layout and hierarchy well, but some interaction flows would have been clearer with clickable prototypes sooner in the process.

The project is still active. New screens still get built from the same kit, and nothing so far has broken the system.
